# MAES Platform v1.2.0

## Summary

`v1.2.0` adds security operations capabilities — UEBA, incident case management, automated playbooks, and threat intelligence enrichment — and fixes several bugs that would have prevented the new modules from loading correctly.

## New Features

### UEBA (User Entity Behavior Analytics)

- Builds behavioral baselines from 30 days of audit log history.
- Detects geographic, temporal, operational, and IP anomalies.
- Computes a composite risk score (0–100) with actionable recommendations (allow, monitor, require MFA, block & investigate).
- Automatically creates high-severity alerts when risk score ≥ 70.
- UEBA Dashboard in the frontend with baseline listing and risk detail panel.

**Endpoints:** `GET /api/ueba/baseline/:userId`, `GET /api/ueba/risk/:userId`, `GET /api/ueba/baselines`, `POST /api/ueba/process-activity`, `GET /api/ueba/stats`

### Case Management (Incident Response)

- Full incident lifecycle: new → investigating → contained → resolved → closed.
- Timeline tracking with user attribution for every status change, assignment, and evidence addition.
- Evidence management with hash verification and chain of custody.
- Incident assignment to investigators.
- Statistics dashboard with counts by status and severity, plus average resolution time.

**Endpoints:** `GET/POST /api/incidents`, `GET /api/incidents/stats/summary`, `GET /api/incidents/meta/playbooks`, `PUT /api/incidents/:id/status`, `PUT /api/incidents/:id/assign`, `POST /api/incidents/:id/evidence`, `POST /api/incidents/:id/playbook`

### Automated Playbooks

Three built-in playbooks with database-backed execution tracking:

| Playbook | Steps | Containment Approval |
|---|---|---|
| Compromised Account Response | 6 | Required |
| Phishing Email Response | 5 | Auto-execute |
| Privileged Access Abuse | 4 | Required |

Playbook execution states: `running`, `completed`, `failed`, `pending_approval`, `error`.

### Threat Intelligence Integration

- Multi-provider IOC enrichment for IPs (AbuseIPDB, Shodan), domains (VirusTotal), and file hashes (VirusTotal).
- Risk level classification: critical, high, medium, low, clean.
- In-memory 1-hour TTL cache with 1000-entry cap.
- Bulk enrichment (1–100 IOCs per request).
- Saved IOCs with CRUD tracking via `/api/threat-intel/saved`.

**Endpoints:** `GET /api/threat-intel/enrich/ip/:ip`, `GET /api/threat-intel/enrich/domain/:domain`, `GET /api/threat-intel/enrich/hash/:hash`, `POST /api/threat-intel/enrich/bulk`, `GET /api/threat-intel/stats`, `GET/POST/DELETE /api/threat-intel/saved`

### Frontend

- Incident Management page with dashboard cards, filterable list, and detail dialog with timeline and playbook actions.
- Threat Intelligence page with single/bulk enrichment and provider status.
- Saved IOCs tab with add/delete/re-enrich functionality.
- UEBA Dashboard page with baseline listing, risk scores, and confidence indicators.
- Behavior Analytics entry in the sidebar.

## Bug Fixes

| Issue | Fix |
|---|---|
| Duplicate `require()` calls and duplicate `const app = express()` in `api/src/index.js` | Removed duplicates; consolidated imports at top |
| Duplicate imports (`Compliance`, `Incidents`) and duplicate routes in `frontend/src/App.jsx` | Removed duplicates |
| Broken `menuItems` array in `frontend/src/components/Sidebar.jsx` | Fixed object literal syntax |
| `pool.query('BEGIN')` used for transactions in `incidentService.js` | Replaced with `pool.connect()` + client-based transactions |
| `updateStatus` recorded `previous_status` equal to `new_status` | Added separate query to capture previous status before update |
| `/stats/summary` and `/meta/playbooks` routes matched by `/:id` first | Moved specific routes before `/:id` parameterized route |
| SQL injection in `getUserCountries()` — days parameter interpolated into query string | Parameterized with `$3` |
| Missing `updateBaseline()` method called by `UebaService` | Added method to `UserBehaviorProfile` |
| UEBA alert creation referenced non-existent `Alert` model import | Replaced with direct `pool.query()` INSERT |
| RBAC permission names (`canViewAlerts`, `canViewSettings`) don't exist in the system | Updated to `canManageAlerts`, `canManageIncidents`, `canAccessThreatIntel`, `canManageSystemSettings` |
| Missing IPQualityScore provider initialization in `iocEnrichment.js` | Added `IPQUALITYSCORE_API_KEY` env var check |

## Security

- All SQL queries use parameterized inputs (no string interpolation).
- All new API endpoints enforce RBAC permissions and rate limiting.
- Transactions use proper database clients with `BEGIN`/`COMMIT`/`ROLLBACK` and `client.release()`.
- No hardcoded credentials or API keys; threat intel providers are optional and fail gracefully.

## Database Migrations

### `007_add_ueba_incidents_playbooks.sql`
New tables: `ueba_baselines`, `incidents`, `incident_timeline`, `incident_evidence`, `playbook_executions`. Alters `alerts` to add `incident_id` column.

### `008_add_saved_iocs.sql`
New table: `saved_iocs` with unique constraint on `(organization_id, value)`.

## Required Environment Variables

All v1.1.0 secrets are still required. New optional variables:

```bash
VIRUSTOTAL_API_KEY=         # Optional — enables VirusTotal enrichment
ABUSEIPDB_API_KEY=          # Optional — enables AbuseIPDB enrichment
SHODAN_API_KEY=             # Optional — enables Shodan enrichment
IPQUALITYSCORE_API_KEY=     # Optional — enables IPQualityScore enrichment
```

## Deployment

1. Pull the release.
2. Update `.env` with threat intel API keys if desired.
3. Start the stack: `docker compose up -d --build`
4. Apply migrations:

```bash
docker compose exec -T postgres psql -U maes_user -d maes_db \
  < database/migrations/007_add_ueba_incidents_playbooks.sql
docker compose exec -T postgres psql -U maes_user -d maes_db \
  < database/migrations/008_add_saved_iocs.sql
```

5. Restart the API: `docker compose restart api`
